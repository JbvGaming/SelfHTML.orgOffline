[Bearbeiten] Allgemeines

Diese Seite gibt Empfehlungen zu einem guten CSS Coding-Stil. Sie behandelt nicht Problemlösungen mittels CSS.

CSS besteht in der Anwendung von kaskadierenden Regeln auf das DOM eines geparsten HTML Dokuments. Allein dieser Mechanismus regt zu einigen Coding-Empfehlungen an.

CSS-Files werden manchmal auch im Teamwork bearbeitet. Dadurch ergibt sich zwangsläufig ein Bedarf nach Style-Guides. Dabei soll geklärt werden, in welcher Reihenfolge Regeln erstellt werden sollen, wie Klammern zu notieren sind und ob es eine Richtlinie für die Reihenfolge der Eigenschaften geben soll. All dies kann die Lesbarkeit des Codes erhöhen.

[Bearbeiten] Organisation

In der Frage, wie man CSS-Styles am besten organisiert, herrscht darin Einigkeit, dass

  • das Einbinden von CSS-Dateien den Notationen in style-Elementen vorzuziehen ist,
  • Notationen in style-Elementen den Inline-Styles (style-Attribut) vorzuziehen sind,
  • falls mit Javascript Styles manipuliert werden, nicht das style-Attribut verändert wird, sondern die Elemente anderen Klassen zugeordnet werden.

Besonders Inline-Styles sind nachteilig, da sie sich nicht medienabhängig spezifizieren lassen.

CSS-Dateien erzeugen zwar einmalig einen zusätzlichen Request pro Datei. Da sie aber in der Regel gecached oder von Bots gar nicht erst angefordert werden, erzeugen sie deutlich weniger Datentransfer.

Es gibt verschiedene Ansichten, wie CSS-Dateien zu organisieren sind. Die einen trennen lieber Print- und Screen-CSS in separate Dateien, andere möchten wiederum alles in einer Datei haben und dadurch Requests sparen. Die einen bevorzugen es, separate Dateien für spezielle Anpassungen an ältere Browser zu schreiben, andere möchten aber die Anpassungen im normalen Kontext sehen.

Generell gibt es keinen Grund, warum man notwendige Anpassungen an MSIE in separate Files schreiben muss, denn Sie haben zwei Möglichkeiten:

  • Sie verwenden Selektoren (oft auch CSS-Hacks genannt), welche nur von einer spezifischen MSIE-Version interpretiert werden.
Beispiel
<style type="text/css"> h1 {color: red;} * html h1 {color: blue;} /* IE 6 */ *:first-child + html h1 {color: green;} /* IE 7 */ </style>
Der CSS-Code ist unsinning, weil das html-Element kein Elternelement hat (* html) bzw. auf kein anderes Element folgt (* + html). Dieser Code wird nur von den Internetexplorern 6 bzw. 7 akteptiert, die ursprüngliche Festlegung überschrieben und die Überschriften erhalten die entsprechenden Farben.
  • Sie weisen mittels Conditional-Comments einem Element eine besondere ID oder Klasse für eine MSIE-Version zu und können diesen Browser dann mittels des Nachfolgeselektors ansprechen.
Beispiel
... <style type="text/css"> #ie6 h1 {color: blue;} #ie7 h1 {color: green;} h1 {color: red;} </style> ... <body <!--[if IE 6]>id="ie6"<![endif]--><!--[if IE 7]>id="ie7"<![endif]-->> <h1>Überschrift</h1> ...
Dieser Code führt letztendlich zu folgendem Quelltext:

im IE6: 
<body id="ie6">
  <h1>Überschrift</h1>
im IE7: 
<body id="ie7">
  <h1>Überschrift</h1>
in allen anderen Browsern: 
<body>
  <h1>Überschrift</h1>

Deshalb erhalten die Überschriften die entsprechenden Farben.

Das Argument, dass man mit Requests sparsam umgehen sollte, ist sicher zu überlegen. Mehrere Dateien sind zudem für das Debugging hinderlich.

[Bearbeiten] Syntax-Stil

Es gibt mehrere Arten, Syntax zu notieren. Wichtiger als die Art ist die Konsistenz. Hier einige Beispiele, beginnend mit einem schlechten Beispiel.

Beispiel 
#someid1 .someclass1:pseudoclass,#someid1 .someclass1:pseudoclass { border:1px solid grey;
margin:0;
  color: #ccc; margin-left: 10px;background;#000 url(some.png);
  display:block;margin-right:20px; 
}
Wir vermissen Konsistenz. Es gibt keine Reihenfolge der Eigenschaften innerhalb eines Blocks. Whitespace ist uneinheitlich notiert. Das Komma zwischen den Selektoren übersieht man fast.
#someid1 .someclass1:pseudoclass, 
#someid1 .someclass1:pseudoclass 
{ 
  border:1px solid grey;
  margin:0;
  color:#ccc; 
  margin-left:10px;
  background;#000 url(some.png);
  display:block;
  margin-right:20px; 
}
Hier wurden die Selektoren je in eine separate Zeile geschrieben.

Die Block-Klammer beginnt auf einer neuen Zeile. Editoren mit Folding können damit umgehen. Whitespace wurde konsistent angewendet.

Aber die Reihenfolge der Eigenschaften ist immer noch im Argen.
#someid1 .someclass1:pseudoclass, 
#someid1 .someclass1:pseudoclass 
{ 
  display:block;
  margin:0 20px 0 10px;
  border:1px solid grey;
  color:#ccc; 
  background;#000 url(some.png);
}
Hier wurde die Reihenfolge umgestellt und zudem die Shorthand-Schreibweise für margin gewählt.
Was ist eine gute Reihenfolge? Ein Vorschlag könnte lauten: Zuerst display und float, dann weitere Blockeigenschaften, dann border-Eigenschaften, dann color- und background-Eigenschaften.

Eine gute Syntax hilft Ihnen, indem Sie den Code schneller lesen können und Sie in einem Block ahnen, wo die Eigenschaft zu erwarten ist. Bestimmte Schreibweisen für Block-Klammern erlauben zudem, dass Editoren ihr Folding anwenden können. Dadurch bleiben auch grössere CSS-Files übersichtlich.

[Bearbeiten] Normalisierung

Browser wenden ihr eigenes Default-CSS an, bevor sie das Autoren-CSS anwenden. Die Defaultregeln können aber von Browser zu Browser abweichen.

Manche Autoren sehen sich dadurch veranlasst, Teile der Elemente zuerst zu normalisieren. Jedoch ist hier Vorsicht geboten. Ganz besonders Formular-Elemente sollte man nicht blindwütig normalisieren.

Beispielsweise wird die Einrückung von Listen durch die Browser unterschiedlich gehandhabt. Die einen realisieren dies über Innenabstände des ul- bzw. ol-Elementes, die anderen über Außenabstände der li. Deshalb ist die Normalisierung vor allem für folgende Elemente verbreitet: ul, ol, li, dl, dt, dd.

Normalisierung sollte vor allen anderen Regeln betrieben werden.

Beispiel 
ul, ol, li, dl, dt, dd {
  display: block;
  padding: 0;
  margin: 0;
}
 
li {
  display: list-item; 
  margin-left: 2em;
}
